# Uddokta Business Blueprint

## Working definition

**Uddokta** is a mobile-first sales, order, follow-up, customer-record, and trust-control system for Bangladeshi Facebook-commerce and WhatsApp-heavy SME sellers.

It is not a chatbot company, not a digital marketing agency, and not a BotSailor reseller. It is a product-led business built by combining free/public/open-source technology into a controlled, branded, Bangladesh-specific operating system for small sellers.

The first commercial promise:

> **Turn a messy Facebook/WhatsApp inbox into a mobile order-control system.**

The product must feel tangible to a Bangladeshi SME owner. They should receive a login, a mobile dashboard, order cards, customer records, auto-replies, follow-up controls, reports, and a trust profile page. They should not see API keys, webhooks, n8n, Chatwoot, Typebot, Supabase, Meta settings, server logs, or technical jargon.

# 1. Strategic thesis

## 1.1 The market problem

Bangladeshi F-commerce and WhatsApp sellers do not primarily suffer from lack of a website. They suffer from **operational chaos**.

The common workflow:

1. Customer comments "price?"
2. Customer sends Messenger or WhatsApp message.
3. Seller replies manually.
4. Product, size, color, address, payment, and delivery details are scattered across chat.
5. Payment proof arrives as screenshots.
6. Order is copied into a notebook, Google Sheet, courier portal, or staff inbox.
7. Follow-up depends on memory.
8. COD return risk is not checked.
9. Reviews are not collected systematically.
10. Owner has no clean report.

This creates two core business failures:

## 1.2 Core problem one: sales leakage

Sales leakage is the hidden loss created by late replies, missed messages, unstructured orders, no follow-up, and staff inconsistency.

Typical seller pain:

- Many inquiries, fewer confirmed orders.
- Staff forgets to reply.
- Staff cannot find old customer conversations.
- Customers ask "available?" and disappear.
- Boosted ad money is wasted because leads are not captured.
- No owner-level visibility into who handled which customer.
- No structured follow-up for interested but undecided buyers.
- No conversion visibility from inquiry to order.

## 1.3 Core problem two: trust and proof failure

Informal Facebook sellers struggle to prove credibility. Buyers fear scams, fake products, fake pages, non-delivery, counterfeit goods, wrong size/color, and refund problems. Sellers also suffer from fake buyers, COD returns, forged payment screenshots, and dispute screenshots spread across Facebook groups.

Typical seller pain:

- No professional invoice.
- No order confirmation record.
- No delivery proof trail.
- No customer purchase history.
- No return/refund ticket.
- No verified review engine.
- No trust profile.
- No structured business record for future growth, suppliers, courier negotiation, or finance discussion.

## 1.4 Uddokta product thesis

Uddokta bundles both problems into one system:

**Sales Control + Trust Control**

The platform gives a seller:

- One mobile dashboard.
- One inbox/control panel.
- One order board.
- One customer database.
- One follow-up system.
- One digital receipt/trust link.
- One weekly performance report.
- One operating process for staff.

The emotional promise:

> "Your business will no longer depend on scattered screenshots, forgotten messages, and staff memory."

The rational promise:

> "Every inquiry becomes a trackable lead, every order becomes a structured card, and every customer becomes a record."

# 3. Ideal customer profile

## 3.1 First wedge segment

The strongest first wedge is **premium F-commerce sellers in cosmetics/skincare and gadgets/accessories**.

Reason:

- High message volume
- Consultative selling
- High buyer skepticism
- High need for product explanation
- Higher average order value
- Repeat purchase potential
- Stronger ability to pay
- Need for customer history
- Need for trust-building
- Need for order tracking
- Need for staff control

## 3.2 Segment ranking

| Segment | Priority | Reason |
|---|---:|---|
| Cosmetics / skincare | Highest | Consultative, repeat-purchase, trust-heavy, high DM volume |
| Gadgets / accessories | Highest | Warranty/trust concerns, high AOV, fake-product fear |
| Clothing / fashion | Medium | Huge volume but low margin, high support burden |
| Education/training | Medium | Good lead capture use case but fewer logistics pains |
| Home decor | Medium-low | Higher AOV but lower daily volume |
| Food/cloud kitchen | Avoid initially | Urgent, low margin, operationally different |
| Generic services | Later | Less repeatable template |

## 3.3 Disqualified early buyers

Avoid early sales to:

- Sellers with no order volume.
- Sellers who only want free AI chatbot.
- Sellers who refuse any monthly fee.
- Sellers who expect 24/7 phone support for a tiny subscription.
- Sellers who want mass WhatsApp spam.
- Sellers who cannot define products, prices, or delivery rules.
- Sellers who sell prohibited/regulated items.
- Sellers who want to connect personal WhatsApp numbers without accepting risk.

- Setup fee creates commitment.
- Monthly fee creates recurring revenue.

- Done-for-you setup: ৳7,999–৳14,999

## 6.3 RLS principle

Every table must have:

```sql
alter table table_name enable row level security;
```

Every read/write policy must check that the authenticated user belongs to the same workspace.

Pseudo-policy:

```sql
create policy "workspace members can read"
on table_name
for select
using (
  exists (
    select 1
    from workspace_users wu
    where wu.workspace_id = table_name.workspace_id
    and wu.user_id = auth.uid()
    and wu.status = 'active'
  )
);
```

# 9. Workflow automation

## 9.1 n8n workflow list

### Workflow A: inbound message router

Trigger:

- Meta webhook
- Evolution API webhook for demos

Steps:

1. Receive message.
2. Identify workspace by channel ID.
3. Normalize sender phone/profile ID.
4. Create or update customer.
5. Create or update conversation.
6. Save message.
7. Check business rules.
8. If auto-reply matches, queue reply.
9. If human needed, notify staff.
10. Log event.

### Workflow B: keyword auto-reply

Triggers:

- price
- দাম
- available
- stock
- delivery
- return
- location
- order
- payment
- warranty
- original
- size
- color

Steps:

1. Detect intent.
2. Check product context if available.
3. Select template.
4. Personalize.
5. Send reply.
6. Mark conversation as bot-handled.
7. Notify staff if high intent.

### Workflow C: create order from chat

Trigger:

- Staff clicks "Create Order"
- Typebot order form completed

Steps:

1. Collect customer info.
2. Collect product/variant/quantity.
3. Calculate total.
4. Create order number.
5. Save order.
6. Generate receipt link.
7. Send confirmation message.
8. Add follow-up reminder.
9. Backup to Google Sheet.

### Workflow D: abandoned inquiry recovery

Trigger:

- Conversation has no order and no reply after defined business rule.

Steps:

1. Check last customer message.
2. Check if staff replied.
3. Check if customer already ordered.
4. Send soft follow-up.
5. Escalate to owner if still no response.
6. Mark as recovered/lost.

### Workflow E: payment proof handler

Trigger:

- Customer uploads screenshot or sends bKash TrxID.

Steps:

1. Attach file/message to order.
2. Mark payment as "Needs verification".
3. Notify staff.
4. Staff marks verified manually.
5. Send payment confirmation.

Do not automate financial confirmation unless using verified payment API access.

### Workflow F: courier workflow

Trigger:

- Order status becomes "Ready to Ship".

Steps:

1. Validate customer phone/address.
2. Prepare courier payload.
3. Submit to courier API or export CSV.
4. Save consignment ID.
5. Send customer tracking message.
6. Add delivery follow-up.

### Workflow G: review collection

Trigger:

- Order marked delivered.

Steps:

1. Send thank-you message.
2. Ask for rating/review.
3. Store review.
4. Ask permission to publish.
5. Add to trust profile if approved.

### Workflow H: weekly report

Trigger:

- Recurring schedule.

Steps:

1. Count inquiries/orders/returns/follow-ups.
2. Calculate estimated leakage.
3. Generate PDF/image summary.
4. Send to owner.
5. Store in reports table.

### Workflow I: onboarding automation

Trigger:

- New paid setup / approved pilot.

Steps:

1. Create workspace.
2. Create owner user.
3. Create Google Sheet backup.
4. Create client folder.
5. Create default templates.
6. Create product import sheet.
7. Send onboarding form.
8. Notify internal admin.

## 9.2 Workflow reliability rules

- Every workflow must be idempotent.
- Every webhook event must store raw payload.
- Every failed message must enter retry queue.
- Every API call must log response.
- Every client must have bot pause switch.
- Every outbound message must record status.
- Every automation must have human override.

# 16. Compliance and risk

## 16.1 WhatsApp risk

Risks:

- Meta policy violation
- Spam reports
- Template rejection
- Number quality downgrade
- Unofficial QR session disconnect
- Message charges
- Business verification friction

Controls:

- Use official API for production.
- Restrict broadcast features.
- Make message categories clear.
- Add opt-out language.
- Require client approval for campaign messages.
- Keep logs.
- Avoid general-purpose AI chatbot positioning.
- Add bot pause switch.

## 16.2 Data privacy

Risks:

- Cross-client data leak
- Staff access abuse
- Exposed API key
- Public receipt leakage
- Unauthorized exports

Controls:

- Supabase RLS
- Server-side workspace checks
- Audit logs
- Role-based permissions
- Signed public links
- Service role only on server
- Data export controls
- Secure environment variables

## 16.3 Business risk

Risks:

- Low willingness to pay
- Too much support burden
- Clients expecting instant sales
- Technical fragility
- Courier/API instability
- Wrong niche
- Too many features too early

Controls:

- Paid setup fee
- Strong qualification
- Avoid low-volume clients
- Start with cosmetics/gadgets
- Keep MVP narrow
- Use templates
- Document support boundaries
- Sell operational control, not magic revenue
